## 今日一句话总结

Campaign Center 新增持久化 Batch Queue：Flow 2-10 正在发送时，可以先准备并确认下一批 Flow 1 新名单；当前批次发送及 Notion 收尾成功后，下一批才会自动开始。

## 从什么变成什么

| 范围 | 修改前 | 修改后 |
|---|---|---|
| 同时安排 Campaign | 一个 Campaign 运行时不能准备另一个 Campaign | 当前 Batch 运行时仍可导入、修改并预览下一批 Flow 1 |
| 下一批启动 | 必须等当前批完成后再回来手动操作 | 用户确认后加入 Queue，当前批安全完成后自动接力 |
| 批次内容 | 等待期间名单可能被后来操作影响 | Queue 保存独立 Run，冻结项目、名单、Sender、模板、模式与时间配置 |
| Notion 安全 | 发送结束与 Notion 更新之间存在短暂空档 | Notion 为 WAITING、RUNNING、FAILED 或 PARTIAL 时都不会启动下一批 |
| 重启恢复 | Server 重启后等待工作可能丢失 | Queue 写入本机 `campaign-data`，重启后仍可恢复 |
| 重复点击 | 可能担心同一批被排两次 | 同一个 Run ID 只会进入 Queue 一次 |
| Queue 操作 | 没有统一状态 | 发送台显示下一批、等待原因，并可取消或人工确认放行 |
| 发送状态 UI | 只能进入 Flow 页面后查看各自进度 | Campaign Center 顶部统一显示 Prepared、Sending、Notion、Next Batch 四阶段和动态真实进度 |

## Queue 现在怎样运行

1. Flow 2-10 正在发送。
2. Agent 切到 Flow 1，导入新 Excel、修改名字、选择 Sender 并生成预览。
3. `开始发送` 会显示为 `加入等待队列`。
4. Agent 仍需确认 LIVE 与 opt-in；Queue 不会绕过发送确认。
5. 当前 Flow 2 完成发送后，Mamba 先推进 Notion Flow 状态。
6. 只有 Notion 收尾成功，Queue 才恢复下一批 Flow 1 并自动启动。
7. 若 Notion 更新失败、部分失败或当前批被手动停止，Queue 会 Hold，不会继续发送。
8. Agent 可取消下一批；人工检查问题后，也可以明确确认放行。

## 安全决定

- 任何时刻仍然只允许一个 Campaign Runner 真正发送，不会同时跑两个 Batch。
- 排队预览不会覆盖当前 `active-run`，避免 Tracker 或状态页读到下一批资料。
- Queue 里的 Run 使用独立状态文件；当前批的名单与下一批名单不会共用可变数组。
- Flow 1 新名单编辑不会改变已经开始发送的 Flow 2 assignments。
- 服务器重启后，打开 Campaign Center 会读取等待 Queue；已经确认过的批次可以继续接力。
- 手动 STOP 不会自动接下一批，因为 STOP 可能代表系统或号码需要检查。

## 手机回复自动核对

- 修改前：客户回复后会进入 Today；如果 Agent 直接从手机 WhatsApp 回复，Mamba 不会马上知道，Today 要等人工操作或早间扫描才会变化。
- 修改后：Campaign Console 开着时，每 30 分钟自动核对负责号码发出的消息。
- 只处理已经收到客户回复、已经进入 Human Takeover、今天到期，而且销售在今天发出、时间晚于客户最新回复的记录。
- 昨天或更早的旧 outbound 不会重复把 Follow-Up 一天一天往后推。
- Running 的自动 Flow 客户会被排除，避免把群发模板误判成 Agent 已跟进。
- 找到手机回复后，`Follow Up At` 自动排到明天 10:00（KL），客户从今天清单移走，明天会再次出现。
- Follow-Up Desk 新增 `Check My Replies`，可立即核对，并显示上次、下次核对时间和本轮移出 Today 的人数。
- 自动核对会写入 System Logs 和本地 Conversation History，失败时页面会显示明确原因。
- 新增完整模拟测试：验证手机 outbound 会更新 Notion、刷新本地 cache；STOP、未来任务和 Running Flow 不会被误处理。

- 新分类遇到 Not Interested、Spam、Wrong Target 等终止路线时会清空 `Follow Up At`，避免 Notion 留下假的预约任务。
- 保留一个只修复终止类漏判的安全重判模式，不需要把所有 Warm 客户重新分类。

## Telegram 客户回复按项目分 Topic

- 找到根因：Tracker 以前只看当前电脑的 active run 和本地 tracker 记录。若另一台电脑负责 blast，这台收件电脑会收到回复，却认不出客户属于 Binastra 或 Enlace。
- Telegram Topic 与收件箱配置本身正常；Binastra、Enlace Topic 均已存在，Topic store 也匹配当前收件箱。
- 修改后先用 active run、tracker 与本机 Blast Leads cache 快速识别；本地查不到时，才会按电话号码和 WhatsApp sender 去 Notion Blast Leads 找客户最新项目。
- 同一个电话号码若有多个项目记录，优先选择 Sender Instance 相符的记录，再以最新 blast 时间决定。
- 只有 Notion 也找不到的陌生号码才会进入「陌生来信」；不会再把跨电脑 blast 的已知客户当成 General/未知客户。
- Tracker console 会明确显示 `phone / project / source`，方便确认这次路由来自 local 还是 Notion。
- 找到另一个 General 来源：Brain 审批草稿以前仍走旧 Telegram sender，没有附上 Topic ID。现在审批按钮卡、AI 失败提示会进入对应项目 Topic，投诉/STOP 提示进入 STOP Topic。
- 本轮只做资料读取与模拟测试，没有发送真实 Telegram 或 WhatsApp 消息。

## 当前限制

- 这次验证使用模拟 Runner 与只读页面检查，没有用真实联系人跑自动接力；真实发送仍建议先用 2-5 个获准测试号码验证一次。
- Queue 保存在当前电脑。不同电脑各自有本机 Queue，不会自动共享同一个等待列表。
- Notion 或 WhatsApp 服务异常时，Queue 会宁愿停住，也不会猜测成功后继续发送。
