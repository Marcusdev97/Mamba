## 修改前发现的问题

- Mamba 同时存在旧 Control Center、Campaign Console、多个 `.command` 和不同端口入口，可能打开不同 UI，也容易重复启动服务。
- Reply Tracker 与 Sales Brain 可能被 launcher、launchd 和 Mamba Server 分别启动，存在抢 webhook 或重复服务的风险。
- Sales Brain 会跟随回复服务启动，但目前自动回复仍未准备进入正式测试。
- 同一个电话号码的多条回复会分别查询 Notion；找不到 Blast Leads 时，每条消息又各自重试。
- Blast Leads 完整缓存可能被重复下载。数据库扩大后，启动、同步和客户页面会越来越慢。
- Reply Tracker 的错误主要出现在 Terminal，System Logs 缺少明确的错误代码、影响和处理建议。
- macOS `Mamba.app` 没有正式 App Icon。

## 今天完成

### 1. 统一 Mamba App 和 UI 入口

- `Mamba.app`、`Mamba Control Center.command` 和网页入口统一进入 `:8787/control-center`。
- 删除旧的独立 `:8810` Control Center 服务和 `campaign-app/control_center.mjs`。
- `/` 不再代表另一套 UI；默认打开统一 Control Center。
- 保留一个主要启动入口，避免从不同 command 打开不同版本的界面。
- README 的架构、目录和服务地址已经更新为统一 Mamba Server。

### 2. 清理重复 launcher 与后台服务

- 删除已经被统一网页取代的旧 launcher：
  - `Campaign Console.command`
  - `Conversations.command`
  - `Follow Up Desk.command`
  - `Live Reply Tracker.command`
  - `Sales Brain.command`
  - `Set Notion Token.command`
  - `选人发下一轮.command`
- launchd 不再常驻启动 Reply Tracker 和 Sales Brain。
- Reply Tracker / Sales Brain 改由 Mamba Server 根据 Settings 统一管理。
- 旧的 `com.mamba.brain` 和 `com.mamba.tracker` LaunchAgent 会在安装维护任务时清理。

### 3. Sales Brain 自动回复安全开关

- 新增 `MAMBA_BRAIN_ENABLED` 安全开关，默认值为关闭。
- Settings 可以显示和保存自动回复开关。
- 关闭时仍会运行 Reply Tracker、记录客户回复、执行 STOP 和 Notion 队列，但不会自动回复客户。
- Daily Campaign Launch Gate 能识别 Tracker-only 模式，不会把“人工关闭 Brain”误报为服务故障。
- Control Center 会明确显示 `Sales Brain (Manual Off)`。

### 4. Mamba macOS App Icon

- 从提供的 Mamba Logo 重制深色圆角 App Icon。
- 生成 1024px PNG 和 macOS 多尺寸 `.icns`。
- `Mamba.app/Contents/Info.plist` 已登记 `Mamba.icns`。

### 5. Notion 回复查询与重试优化

- 新增按电话号码协调的 Notion 回复队列。
- 同一个号码的多条消息共享一次客户查询，但每条回复仍按 message ID 独立保存和处理。
- 渐进重试间隔为 15 秒、30 秒、1 分钟，之后逐步增加，最长每小时一次。
- 新消息会加入同号码现有的重试时间，不再立即制造另一轮查询。
- STOP 类回复绕过冷却并优先检查，避免停止联系指令被延迟。
- 普通回复等待超过 24 小时仍找不到 Blast Leads 时，会转为人工检查并停止持续消耗 Notion 请求。
- 手动 Push 可以重新处理人工检查项目，并继续清空同号码剩余队列。
- 队列状态、错误代码、帮助说明、下次重试时间和人工检查状态会持久化，重启后不会丢失。

### 6. Blast Leads 缓存防重复

- 同时发生的完整数据库读取会共用同一个请求。
- 自动同步在缓存 10 分钟内会复用本地资料。
- 用户手动 Sync 可以绕过 10 分钟缓存，但 30 秒内的重复点击会被合并。
- 缓存改用临时文件加 rename 写入，避免中途退出留下损坏 JSON。
- 如果分页读取失败，会保留旧缓存，不会把不完整的客户资料覆盖进去。

### 7. 更清楚的 Error / System Logs

- Notion 回复问题现在使用固定错误代码，例如：
  - `NOTION_AUTH_FAILED`
  - `NOTION_DATABASE_ACCESS_FAILED`
  - `NOTION_RATE_LIMITED`
  - `NOTION_NETWORK_FAILED`
  - `NOTION_LEAD_NOT_FOUND`
  - `NOTION_REPLY_MANUAL_REVIEW`
  - `NOTION_RETRY_COORDINATOR_FAILED`
- 每条错误会说明：
  - 发生了什么
  - 客户回复有没有丢失
  - 影响哪个功能
  - 系统是否会自动重试
  - 下次重试时间
  - 建议怎样解决
- Reply Tracker 会直接把结构化问题写进 System Logs。
- Control Center 会显示等待 Notion 同步的消息数、电话号码数量和人工检查数量。

### 8. Device Ownership 补救 Dry Run

- 新增 `campaign-app/device_ownership_repair.mjs`，用来检查旧资料可以归属到哪一台电脑与哪一个 WhatsApp 号码。
- 唯一连接身份采用 `Device ID + WhatsApp sender phone`；`wa_01`、`wa_02` 只作为 instance 名称，不当作跨电脑唯一键。
- 只有同时记录明确 Device ID、sender phone 和成功发送时间的 campaign run 才能作为归属证据。
- 旧 run 没有 Device ID 时一律忽略；“文件出现在本机”不能证明是本机发送。
- WhatsApp 历史 outbound 只能证明 sender phone，不能证明是哪台电脑发送，因此不会用来自动补 Device ID。
- 新 blasting 的 run 会保存 Device ID；写入 Notion 的 sender key 改为 `Device ID::sender phone`，不再使用可能在不同电脑重复出现的 `wa_01` / `wa_02`。
- 旧 run 之后即使在另一台电脑恢复或上传，也不会自动继承当前电脑的 Device ID。
- 同一客户有多个项目、多个发送号码或证据不足时会标成冲突/无法确认，不会猜测。
- 已有 Device Ownership 的资料不会被覆盖。
- 当前工具只支持 `--dry-run`，明确拒绝 `--apply`；不会修改 Notion、不会发送 WhatsApp、不会启动 AI 回复。
- 报告保存在 Git 已忽略的 `campaign-data/device-ownership/`，因为报告包含客户号码和 Notion page ID，不应提交到 Git。

### 9. Customer Desk 严格 Device Scope

- Inbox、Follow-Up、对话历史、人工修改、Refresh Replies 和 Classify Replies 全部只接受具有完整 `Device ID::sender phone` 归属的客户。
- 旧的 `Sender Instance = wa_01 / wa_02` 不再视为本机证据；没有完整 Ownership 的资料会隐藏。
- Follow-Up 每 30 分钟后台核对也使用同一套 Device Scope；本机没有客户时不会扫描 WhatsApp history。
- API 会返回 shared/local/hidden 数量，让页面明确显示有多少共享资料被隐藏，而不是只在前端藏起 rows。
- Customer Desk 会复用 10 分钟内的 Blast Leads cache，避免每次打开页面都重新下载完整 Notion database。

## 已知行为与注意事项

- 修改 Reply Tracker、Sales Brain 或 Settings 后，需要重新打开 Mamba 才会完全加载新服务配置。
- 自动回复仍然默认关闭；今天没有开启 AI 自动回复。
- 转为人工检查的回复不会被删除。如果后来在 Notion 建立了客户资料，可以从 Reply Tracker 手动 Push。
- STOP 回复不会因为超过 24 小时而被静默放弃。
- Device Ownership 默认不扫描 Evolution Chat，因为 Chat 历史不能证明发送电脑；需要检查 sender phone 观察资料时可另外使用 `--scan-chat-observations`，但仍不会据此确认 Device ID。
- 当前 `MAMBA_DEVICE_ID` 尚未固定，正式提供 Apply 功能前必须先在每台电脑设定不会改变且不重复的 Device ID。
- 这次包含删除旧 launcher；提交前应检查 `git status` 和 staged diff，确认这些删除符合预期。
