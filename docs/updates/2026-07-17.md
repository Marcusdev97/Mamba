# Mamba Update — 2026-07-17

## Local Database Shell

- 选择 SQLite 作为每台 Mac 的本机运行数据库；目前不使用 MySQL，也不修改 Evolution 自己的 PostgreSQL。
- 新增 `campaign-data/mamba.sqlite` 数据库壳，文件属于本机运行资料并由现有 `.gitignore` 排除。
- Schema v1 包含 Device、Sender Account、Customers、Conversations、Messages、Operations、Ownership Changes、Sync Jobs 和 Import Runs。
- `sender_accounts` 强制一个 Device 只有一个真实 Sender Phone，同时同一个 Sender Phone 不能静默属于另一台 Device。
- `ownership_changes` 保存修改前后值与重试状态，为之后按 `operation_id` 精确 Rollback 做准备。
- `sync_jobs` 预留 Notion 双向同步、幂等键、失败重试和错误说明；今天没有启动任何同步 worker。
- Settings 新增 Local Database 卡片，可查看 SQLite driver、Schema、文件大小、Device/Sender 和数据数量。
- “初始化 SQLite 数据库壳”只创建本地空表与约束，不读取 Notion、不发送 WhatsApp、不改变 Customer Desk。
- “Notion → SQLite”按钮暂时禁用，等完整 Dry Run、数量对账和 Device Scope 检查完成后才会开放。

## Database Choice

- SQLite 适合当前“一台 Mac 对应一个 WhatsApp 号码”的运行方式：无需常驻数据库服务、离线可用、备份简单。
- 如果未来需要多台电脑实时编辑同一批客户，再迁移到独立的 Mamba PostgreSQL；不建议把 Mamba 表写进 Evolution 的数据库。

## Smart Campaign Scheduling

- 结束时间留空时改为 `AUTO`：按本批实际会发送的人数与安全发送间隔，自动计算尽早完成时间，不再默认拖到晚上 9 点。
- 手动填写 `21:00` 或其他结束时间时才使用 `FIXED`，把客户平均铺在指定时间窗内，并为最后一位客户的 Part 2 预留时间。
- 排队中的 `AUTO` batch 轮到真正启动时，会按当下时间重新排程；不会沿用已经过期的预览时间，也不会突然密集补发。
- Auto 仍保留至少 120 秒的客户间安全间隔与小幅随机延迟；实际完成时间可能受 WhatsApp API、回复检查和网络速度影响而稍晚。
- Flow 1 与 Flow 2–10 两个发送界面都已更新提示，预览会明确显示「自动尽早完成」或「指定时间窗」。
- 新增排程测试，并完成全部 30 个 `test_*.mjs` 测试文件；没有执行 LIVE 发送或写入 Notion 客户资料。

## AUTO 多 Part 排程修正

## SQLite v4 Account / Device Binding Foundation

- 新增 v4 additive schema：`whatsapp_accounts`、`device_bindings`、`send_claims`、`send_events`、`handoff_transfers`、`handoff_log`；现有 v3 正式库不会被原地覆盖。
- 新增 `migrate_v3_to_v4.mjs`；默认只做 Dry Run，`--apply` 也只生成独立的 `mamba.v4.sqlite` 候选库，完成 `quick_check` 与外键检查后才保留输出。
- Sender 身份拆成稳定 WhatsApp Account、Device Binding 和本机 Evolution Connection；`wa_01` 继续只作本机名称。
- `send_claims` 使用 `idem_key` 原子占位；抢锁必须先提交才可调用 Provider。TIMEOUT/过期 CLAIMED 会进入 UNKNOWN 人工检查，不会自动重发。
- 明确区分 `FAILED_RETRYABLE`（Provider 确认完全没发出）与 `UNKNOWN`（不能确认）；只有前者允许受控重试。
- `send_events` 为 append-only 审计，并限制同一 `idem_key` 只能有一个 SENT，Daily Cap 按 MYT 当天唯一 contact 与消息条数计算。
- Cooperative Handoff 使用 `PREPARING → EXPORTED → IMPORTED → COMPLETED` 状态机；Bundle 验证成功前来源 Binding 不释放，导出后不可在来源电脑直接撤销。
- Bundle 使用 scrypt 派生密钥与 AES-256-GCM 认证加密，包含完整 claims/events、原 campaign run ID、STOP、客户归属和 pending sync；错误密码、篡改及过期文件都会拒绝。
- Phase 1 仍不开放 Emergency Takeover，也不声称能硬阻止用户违规双开；跨电脑强制单活留到有原子 Arbiter 的 Phase 2。
- 新增并发、失败重试、Timeout、崩溃恢复、Cap、加密与交接测试；当前发送热路径和 v3 Customer Desk 保持不变。

## Control Center 统计与 Multi-Part Resume 修正

- 修复 Campaign snapshot 没带 `instances` 与 `updatedAt`，导致 Control Center 明明由 `wa_01` 发送却显示 `No sender`、更新时间为空的问题。
- Control Center 会从 `instances`、assignment 和 `perSender` 三处安全恢复 sender；不会因为 compact snapshot 少字段就显示错误号码状态。
- 修复中断在 `WAITING_PART3` / `SENDING_PARTn` 的客户无法 Resume；这些状态现在属于可恢复任务，只继续尚未完成的 Part。
- Resume 不会重发已完成的 Part 1 / Part 2，也不会重发 `SENT`、`FAILED` 或已跳过的其他客户。
- 存在未完成 Part 时不再显示 `COMPLETED`，Control Center 会显示 `INCOMPLETE`、Fully sent、Failed 与 Pending，并按 processed 而不是只按 SENT 计算进度。
- 首页「今日发送」改为「今日已接触客户」，避免把“至少发送过一个 Part 的客户数”与“完整发送全部 Parts 的客户数”混为一谈。
- Control Center 遇到过期 customer cache 会自动走正常同步；SQLite Primary 时会明确显示 `SQLite Customer Store`。
- AI 自动回复关闭时，旧 Brain approval backlog 不再计入首页当前工作队列；不会删除历史 pending 文件。
- 今日错误数字改为读取完整当日日志，并明确标注为 `error log entries`，避免误解为当前仍有同样数量的在线故障。

## Flow Map · Current Project Design

- 旧 Flow Map 原本把 Mamba Brain 与 Notion 画成整个系统中心，已经不符合目前实际运行方式；现改成 Outbound、Inbound、Data Layer、Account / Device、Flow Sequence 与 Safety Rules 六个部分。
- Outbound 明确显示 Operator → Mamba → Safety Gates → Sender Queue → Multi-Part Runner → Evolution → Customer，并说明 Resume 只补未完成 Part、Timeout 不自动重发。
- Inbound 明确显示 Customer Reply → Evolution → Reply Tracker → Local History → Rule Classifier → Human Follow-Up；AI 自动回复保持 OFF。
- Data Layer 分开标示当前真相源与目标真相源，避免把 run JSON、SQLite Primary、Notion 过渡路径和 v4 candidate 混成同一阶段。
- 新增 Account / Device 三层身份：真实号码是 Account，`Device::号码 + generation` 是 Binding，`wa_01` 只是本机 Evolution Connection。
- 新增 PC A → 加密 Bundle → PC B 的 Cooperative Handoff 图，并明确 v4 核心服务虽已实现，发送热路径、Handoff UI 与 Emergency Takeover 尚未开放。
- Flow 1–10 使用真实自动顺序 Day 0 → 18；Flow 5 Furnished List 与 Flow 9 Rental 单独标成条件式人工 Flow。
- 顶部状态会实时读取当前 Device、OPEN WhatsApp、SQLite 健康/模式/客户数、客户范围和 Brain 开关；数据库异常时不再用 `shadow · 0` 假装成正常状态。
- 完成桌面、平板与手机宽度检查，并新增 Flow Map 架构回归测试；完整 `test_*.mjs` 测试全部通过。
